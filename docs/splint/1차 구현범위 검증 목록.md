# 우선 검증할 기술 과제

## 1. 검증 우선순위

| 우선순위 | 기술 과제                 | 영향 화면    | 핵심 의존성                                         |
| ---- | --------------------- | -------- | ---------------------------------------------- |
| P0-1 | 파일 업로드·검증 가능 여부       | S01, S09 | 파일 파서, 임시 저장소, 업로드 API                         |
| P0-2 | MCP 계약 유형 분석 연동       | S02, S03 | `assess_contract_scope`, `list_contract_types` |
| P0-3 | 비동기 검토 작업·상태 관리       | S04, S09 | `review_id`, 작업 실행, 상태 저장                      |
| P0-4 | MCP progress → SSE 전달 | S04      | MCP progress, SSE, 상태 조회 API                   |
| P0-5 | 검토 결과 DTO 정규화         | S05, S06 | `review_contract_candidates` 응답                |
| P0-6 | 오류·재시도·중복 실행 방지       | S04, S09 | 오류 매핑, 멱등성, 재시도 정책                             |

## 2. P0 기술 검증 과제

### P0-1. 파일 업로드·검증

**확인할 내용**

* 지원 확장자 파일을 실제로 수신할 수 있는지
* 최대 파일 크기
* MIME과 확장자 불일치 검출 가능 여부
* 손상·암호화·빈 문서 구분 가능 여부
* `file_path` 또는 `file_content + file_name` 입력 계약
* 실패 시 MCP 호출 전 차단되는지
* 완료·실패 후 임시 파일 삭제 여부

**완료 기준**

* 정상 파일 1건이 MCP 입력까지 전달된다.
* 지원하지 않는 파일과 손상 파일이 MCP 호출 전에 차단된다.
* 화면에 표시할 오류 코드를 구분할 수 있다.

지원 형식의 실제 실행 동작, 손상·암호화 파일, 입력 XOR 계약은 명세만으로 확정되지 않으므로 런타임 통합 검증이 필요합니다.

---

### P0-2. 계약 유형 분석·선택 흐름

**확인할 내용**

* `assess_contract_scope` 실제 응답 구조
* `suggested_contract_type`
* `candidates` 정렬 순서
* 후보가 없거나 동점인 경우
* `IN_SCOPE`
* `CONTRACT_TYPE_UNCERTAIN`
* `OUT_OF_SCOPE`
* `EMPTY_DOCUMENT`
* 사용자가 선택한 계약 유형 저장
* 범위 외 계속 진행 확인 저장

**완료 기준**

* 네 가지 범위 판별 결과를 화면 상태로 변환할 수 있다.
* 사용자 선택 전에는 검토 API가 호출되지 않는다.
* 범위 외 상태에서는 재확인 전 검토가 시작되지 않는다.
* 추천 유형과 최종 선택 유형이 별도 필드로 유지된다.

화면 흐름은 계약 유형 선택과 범위 외 재확인을 검토 시작의 선행 조건으로 사용합니다.

---

### P0-3. 검토 작업·세션 상태 관리

**확인할 내용**

* `review_id` 생성 시점
* 검토 요청과 파일의 연결 방식
* 서버리스 요청 종료 후에도 작업이 유지되는지
* 작업 상태 저장 방식
* 동일 요청 중복 실행 방지
* 새로고침 후 상태 복구
* 결과 저장 범위와 TTL
* 실패 후 같은 작업을 재시도할지 새 작업을 생성할지

**최소 상태**

```text
TYPE_SELECTION_REQUIRED
OUT_OF_SCOPE_CONFIRMATION_REQUIRED
QUEUED
REVIEWING
COMPLETED
REUPLOAD_REQUIRED
FAILED
EXPIRED
```

**완료 기준**

* 검토 시작 API가 즉시 `review_id`를 반환한다.
* 새로고침 후 동일 `review_id`의 상태를 조회할 수 있다.
* 동일 검토가 중복 실행되지 않는다.
* 완료·실패·만료 상태를 구분할 수 있다.

MCP 응답에는 애플리케이션의 `review_id`, 시작·완료 시각과 같은 세션 메타데이터가 없으므로 백엔드에서 별도로 관리해야 합니다.

---

### P0-4. MCP progress와 SSE 연동

**확인할 내용**

* MCP가 실제로 보내는 progress 이벤트 형식
* 이벤트 발생 단계
* `current`, `total` 제공 여부
* 완료 이벤트 존재 여부
* 오류 발생 시 마지막 이벤트 형태
* 이벤트 순서 보장 여부
* SSE 연결 종료 조건
* 브라우저 재연결 가능 여부
* SSE 실패 시 상태 조회 폴백
* 이전 검토 이벤트가 새 검토에 섞이지 않는지

**백엔드 정규화 형식**

```json
{
  "review_id": "rev_01J...",
  "sequence": 4,
  "stage": "CLAUSE_REVIEW",
  "current": 7,
  "total": 17,
  "percent": 41,
  "message": "계약 조항을 비교하고 있습니다."
}
```

**완료 기준**

* MCP 이벤트 한 건이 브라우저 SSE까지 전달된다.
* `review_id`와 `sequence`를 통해 잘못된 이벤트를 무시할 수 있다.
* SSE가 끊어져도 상태 조회 API로 완료 또는 실패를 확인할 수 있다.
* 진행률이 역행하지 않는다.

MCP 명세에는 progress 사용이 언급되지만 이벤트 필드와 완료·오류 알림 형태는 구조화되어 있지 않습니다. 따라서 실제 이벤트 샘플 검증이 필수입니다.

---

### P0-5. 검토 결과 DTO 정규화

**확인할 내용**

* `review_contract_candidates` 실제 성공 응답
* 각 오류 상태에서 배열 값
* `clause_results`
* `missing_standard_clauses`
* `match.status`
* `match.standard`
* `toxic_patterns`
* null과 빈 배열 처리
* 표준조항의 `clause_id`, `category`, `version`
* 사용자 조항 식별자 생성 방법

**완료 기준**

* `NONE`, `EXTRA`, `NO_MATCH` 결과를 사용자 조항 목록으로 표시할 수 있다.
* MISSING을 별도 체크리스트로 표시할 수 있다.
* `toxic_patterns`를 결과 코드와 독립적으로 표시할 수 있다.
* 최상위 상태가 `OK`가 아닐 때 결과 배열을 해석하지 않는다.
* 빈 결과를 “문제 없음”으로 표시하지 않는다.

NO_MATCH는 사용자 조항 결과이고 MISSING은 표준조항 기준의 별도 후보이므로 동일한 배열로 합치면 안 됩니다.

---

### P0-6. 오류·재시도·멱등성

**확인할 내용**

* MCP 오류가 예외인지 상태 응답인지
* `CORPUS_UNAVAILABLE`
* `INVALID_CONFIG`
* `PIPELINE_ERROR`
* 타임아웃
* 네트워크 오류
* 오류별 재시도 가능 여부
* 재시도 시 같은 `review_id` 사용 여부
* 중복 요청 차단 방식
* 실패한 요청의 임시 파일 정리

**완료 기준**

* 오류가 공통 API 오류 코드로 정규화된다.
* 재시도 가능한 오류에서만 재시도 버튼을 제공한다.
* 같은 요청을 연속으로 전송해도 MCP가 중복 실행되지 않는다.
* 내부 오류와 스택 트레이스가 사용자에게 노출되지 않는다.

MCP에는 `retryable`, `review_id`, 멱등성 입력이 없으므로 재시도와 중복 실행 방지는 애플리케이션 계층에서 구현해야 합니다.

---

### P1-4. 임시 데이터·세션 만료

**확인할 내용**

* 임시 파일 저장 위치
* 결과 스냅샷 저장 위치
* 세션 TTL
* 정상 완료 후 삭제 시점
* 실패 후 삭제 시점
* 세션 만료 후 삭제
* 서버 재시작 시 잔존 파일 정리
* 로그와 APM에 계약서 본문이 남지 않는지

**완료 기준**

* 완료·실패·만료 테스트 후 임시 데이터가 제거된다.
* 다른 세션에서 파일과 결과에 접근할 수 없다.
* 운영 로그에 계약서 본문과 LLM 프롬프트가 남지 않는다.

---

## 4. 우선 실행 순서

```text
1. 실제 파일 1건으로 MCP 호출 성공
2. 계약 유형 분석 응답 확인
3. review_id 기반 비동기 작업 생성
4. MCP progress 이벤트 캡처
5. SSE 전달과 상태 조회 폴백
6. 전체 검토 응답 DTO 정규화
7. 결과 목록·MISSING 체크리스트 표시
8. TTL·삭제·세션 격리 검증
```

## 5. 1차 기술 검증 범위

가장 먼저 다음 수직 흐름 하나를 완성합니다.

```text
PDF 업로드
→ 파일 검증
→ 계약 유형 분석
→ 사용자 유형 선택
→ review_id 생성
→ MCP 전체 검토
→ progress SSE 전달
→ 완료 상태 저장
→ 결과 목록 표시
```

이 흐름이 성공해야 S01부터 S05까지의 P0 화면 구현 가능 여부가 확정됩니다. 챗봇과 협의 문구는 이 흐름에서 생성된 결과 DTO와 조항 식별자에 의존하므로 이후에 검증합니다.
